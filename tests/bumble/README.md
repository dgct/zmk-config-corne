# Bumble-based BLE integration tests for ZMK firmware

Passive HID-capture harness for the dgct/zmk fork. **Phase 1**: subscribe
to the keyboard's HOG service as a Bumble central, capture all notifies
as JSONL, diff against archived golden traces. No fault injection yet.

## Why

Catches regressions in the HOG mouse pipeline (the `ENOMSG`-class bug we
shipped a fix for in `dgct/zmk` main; see commit `d9661ee2eab6` for the
bad-firmware reference point) without needing to manually scrub a Mac for
"feels stuck" symptoms. The harness pairs once, then on every replay it
walks the same input scenario and asserts the report stream matches the
golden capture (modulo timestamps).

## Hardware requirement

macOS' CoreBluetooth does not expose raw HCI. You need a USB BT controller
(e.g. Asus BT400, Cambridge Silicon Radio CSR8510) that Bumble can drive
directly via libusb. Plug it in, set `BUMBLE_HCI` (e.g. `usb:0`).

## Setup

```sh
uv venv && source .venv/bin/activate
uv pip install -r requirements.txt
export BUMBLE_HCI=usb:0
export ZMK_BLE_ADDR="DA:30:74:88:DA:30"   # set to your keyboard's BLE address
uv run pytest -v
```

## Layout

```
tests/bumble/
  conftest.py         # bumble device fixture, pair-once cache
  test_passive_capture.py  # phase 1 tests (passive)
  golden/             # archived traces (git-tracked, regenerated by --update-golden)
  requirements.txt
  README.md           # this file
```
